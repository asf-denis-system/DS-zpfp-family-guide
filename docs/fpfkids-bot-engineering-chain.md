# Бот-модуль fpfkids: инженерная цепочка

> Полная цепочка проектирования бота-помощника наставника.
> Источники: PACK-kids-learning (домен), DS-zpfp-family-guide (руководство, карточки, промпты), DS-personal-bot (каркас).

## Архитектурные решения (до цепочки)

| Решение | Выбрано | Обоснование |
|---------|---------|-------------|
| Репо | **Отдельный бот-репо** (DS-zpfp-family-bot) | Другой тип пользователя, другой lifecycle, другой контур L4→L3 |
| Пользователи MVP | Автор + 2-3 тестера из пилота | Достаточно для обратной связи, минимум рисков |
| Фреймворк | aiogram 3.x (как DS-personal-bot) | Уже знаем, деплой на том же VPS |
| AI backend | Claude API (Sonnet для генерации, Haiku для оценки) | Как в AIST planner.py |
| Хранение | SQLite (семья, дети, логи, Mastery Map) | Для MVP достаточно |
| Контент | Из PACK-kids-learning + cards/ из DS-zpfp-family-guide | Source-of-truth = Pack |

## 0. Выделение объектов внимания

Что именно мы характеризуем, измеряем и улучшаем?

### 0.1. Объекты в контуре «бот помогает наставнику»

| # | Объект | Что это | Кто наблюдает |
|---|--------|---------|---------------|
| O1 | **Семейный профиль** | Контекст семьи: дети, наставник, со-наставники, время, ограничения | Бот (хранит), наставник (вводит) |
| O2 | **Стратегия развития** | Маршрут на 6-12 мес: принципы, трек, темп, роли | Бот (генерирует), наставник (утверждает) |
| O3 | **Программа на месяц** | Принцип в фокусе, контексты, частота, кто ведёт | Бот (предлагает), наставник (адаптирует) |
| O4 | **Скрипт эпизода** | Готовый рецепт: фразы, шаги, лог-шаблон | Бот (генерирует по принципу + зоне + треку) |
| O5 | **Лог эпизода** | Что сделали, что ребёнок сказал, сам/подсказка, трудность | Наставник (вводит), бот (хранит и анализирует) |
| O6 | **Mastery Map** | Уровень 0-3 по текущему принципу, контексты переноса | Бот (строит из логов), наставник (подтверждает) |
| O7 | **FM-алерт** | Детекция failure modes по паттернам в логах | Бот (детектирует), наставник (чинит) |
| O8 | **Рефлексия наставника** | Что получилось, где сорвался, что меняет | Наставник (пишет), бот (подсказывает вопросы) |

### 0.2. Объекты бота как инструмента

| # | Объект | Что характеризуем |
|---|--------|-------------------|
| B1 | **Качество скрипта** | ZP-инструктированность (содержит ядро принципа, а не попсу) |
| B2 | **Качество анализа лога** | Корректность определения уровня, FM-детекция |
| B3 | **Качество стратегии** | Реалистичность (учитывает ограничения семьи) |
| B4 | **Полезность бота** | Снижает ли когнитивную нагрузку наставника |
| B5 | **Безвредность** | Не ломает ли агентность через наставника, не генерирует ли давление |

## 1. Характеризация (какие свойства измеряем)

### 1.1. Профиль результата ребёнка (через бота)

Бот не взаимодействует с ребёнком напрямую. Он видит ребёнка через логи наставника.

| Характеристика | Как бот видит | Источник данных |
|---------------|--------------|----------------|
| Уровень по принципу (0-3) | Паттерн в логах: подсказка? знакомый/новый контекст? спонтанно? | O5 (логи) |
| Перенос | Количество контекстов, где Level 3 наблюдён | O5, O6 |
| Агентность | «Сам предложил» vs «ждал подсказку» в логах | O5 |
| Работа с ошибкой | «Расстроился» vs «попробовал ещё» в логах | O5 |

### 1.2. Профиль работоспособности контура

| Характеристика | Как бот видит | Нормальный диапазон |
|---------------|--------------|---------------------|
| Ритм (эпизодов/неделю) | Счётчик логов | 2-5 |
| Регулярность | Перерывы между логами | ≤7 дней |
| Вариативность контекстов | Уникальные контексты в логах | ≥2 за цикл |
| Качество логов | Есть ли цитата ребёнка, есть ли «что менять» | Да/нет |
| Рассинхрон взрослых | Разные стили в логах от разных наставников | — |

### 1.3. Профиль наставника

| Характеристика | Как бот видит |
|---------------|--------------|
| Уровень в МИМ | Вводит при онбординге → определяет глубину поддержки |
| Качество вопросов | Лог: «спросил» vs «объяснил» |
| Рефлексия | Заполняет ли рефлексию, видит ли свои ошибки |
| Подсказки | Доля эпизодов «с подсказкой» (если растёт → FM-05) |

### 1.4. Профиль бота

| Характеристика | Как измеряем |
|---------------|-------------|
| ZP-инструктированность | Скрипт содержит детский вопрос, различение, ложную форму принципа |
| Адекватность зоне и треку | Не даёт школьное задание дошкольнику |
| Реалистичность стратегии | Учитывает бюджет времени и ограничения из профиля |
| Полезность | Наставник использует скрипт / отклоняет |
| Безвредность | Не генерирует давление, не подменяет наставника |

## 2. Проблематизация (какие проблемы решаем)

| Проблема | Для кого | Как бот помогает | Как узнаём, что решена |
|----------|---------|-----------------|----------------------|
| Наставник не знает, какой эпизод провести | Наставник | Генерирует скрипт по принципу + контексту + уровню | Наставник проводит ≥3 эп/нед |
| Наставник не видит прогресс | Наставник | Строит Mastery Map, показывает тренды | Наставник может назвать уровень ребёнка |
| Нет защиты от карго-культа (FM-01) | Контур | FM-detection по паттернам в логе | FM-01 не активен |
| Нет Guard Policy | Контур | Автоматическая проверка: Level 3, ≥3 раз, ≥2 контекста | Переходы по данным, не по календарю |
| Когнитивная перегрузка наставника | Наставник | Готовые скрипты, шаблоны логов, напоминания | Наставник тратит <10 мин/эпизод на подготовку |
| Рассинхрон взрослых (FM-03) | Семья | Памятки для со-наставников, единый профиль | Со-наставник ведёт по тому же принципу |
| Наставник не учится (FM-05) | Наставник | Правило «1 принцип впереди», вопросы на понимание | Наставник может объяснить принцип своими словами |

## 3. Нормирование (что считать нормой)

### 3.1. Нормы ребёнка (через логи)

| Показатель | Норма | Жёлтая зона | Красная зона |
|-----------|-------|-------------|-------------|
| Уровень после 8-12 эпизодов | Level 2+ | Level 1 стабильно | Level 0 |
| Переносов за цикл (4 нед) | ≥3 в ≥2 контекстах | 1-2 в 1 контексте | 0 |
| Доля «сам» в логах | ≥50% | 30-50% | <30% |

### 3.2. Нормы контура

| Показатель | Норма | Жёлтая | Красная |
|-----------|-------|--------|---------|
| Эпизодов/неделю | 3-5 | 2 | <1 |
| Перерыв без логов | ≤7 дней | 8-14 дней | >14 дней |
| Качество логов (есть цитата + «что менять») | ≥80% логов | 50-80% | <50% |
| FM активных одновременно | 0-1 | 2 | ≥3 |

### 3.3. Нормы бота

| Показатель | Норма | Проблема |
|-----------|-------|---------|
| Скрипт используется наставником | ≥70% | <50% → скрипты неадекватны |
| Стратегия принята без серьёзных правок | ≥60% | <40% → не учитывает контекст |
| FM-детекция ложноположительных | <20% | >30% → алерт fatigue |
| Наставник просит «объясни иначе» | <10% скриптов | >20% → непонятный язык |

## 4. Индикаторизация (панель управления)

### 4.1. Панель наставника (видит в боте)

| Индикатор | Частота | Откуда | Действие |
|-----------|---------|--------|----------|
| Текущий принцип + уровень | Всегда | Mastery Map | Ориентир |
| Эпизодов на этой неделе | Еженедельно | Счётчик логов | <2 → напоминание |
| FM-алерт | При детекции | Анализ логов | Конкретное действие |
| Guard Policy: готов к переходу? | После 8+ эпизодов | Mastery Map | Да/нет + обоснование |
| Рефлексия | Раз в неделю | Запрос бота | 3 вопроса |

### 4.2. Панель авторов/методиста (агрегат по всем семьям)

| Индикатор | Частота | Что показывает |
|-----------|---------|---------------|
| Средний ритм по семьям | Еженедельно | Здоровье контура |
| Распределение по уровням | Ежемесячно | Где застревают |
| Топ-3 активных FM | Ежемесячно | Системные проблемы |
| % скриптов использованных | Ежемесячно | Качество бота |
| Dropout rate (бросили) | Ежемесячно | Удержание |

## 5. Скоринг (как агрегировать)

### 5.1. Score ребёнка (по принципу)

Level 0-3 по Mastery Map. Не агрегируем по всем принципам в одну цифру — профиль важнее суммы.

### 5.2. Score контура (светофор)

| Зона | Критерий |
|------|---------|
| Зелёная | ≥3 эп/нед, логи качественные, FM ≤1, ритм ≥3 недели | 
| Жёлтая | 2 эп/нед или логи слабые или FM = 2 или перерыв 8-14 дней |
| Красная | <1 эп/нед или FM ≥3 или перерыв >14 дней или наставник не рефлексирует |

### 5.3. Score бота (NPS-подобный)

После каждого скрипта: «Провели по этому скрипту? Помог?» → Да / Частично / Нет.
Раз в месяц: «Что бот делает хорошо? Что раздражает?» → текст.

**Анти-Гудхарт:** Score ребёнка не оптимизируем — используем для диагностики. Score контура — для управления. Score бота — для улучшения инструмента.

## 6. Интерпретация причин отклонения

**Критический шаг.** Между скорингом и выбором вмешательства. Один и тот же симптом — разные причины.

| Симптом | Возможные причины | Как бот помогает разобраться |
|---------|-------------------|------------------------------|
| Нет переноса (Level ≤2 после 12 эп) | Мало контекстов? Наставник подсказывает? Принцип слишком абстрактен? | Показать: сколько контекстов было? Доля «с подсказкой»? |
| Ребёнок отказывается | FM-04 (перегруз)? Усталость? Давление? Скучно? | Спросить: когда отказывается — вечером? После чего? |
| Наставник не ведёт логи | Нет времени? Не понимает зачем? Формат сложный? | Упростить ввод лога (голосовое? 3 кнопки?) |
| Guard Policy не проходит | Мало эпизодов? Мало контекстов? Ребёнок на Level 1? | Показать Mastery Map + конкретное предложение |
| Ритм падает | Болезнь? Отпуск? Выгорание наставника? FM-05? | Мягкое напоминание + вопрос «что произошло?» |

**Правило бота:** Один симптом → показать 2-3 возможные причины → спросить наставника → только потом предложить действие. Не прыгать к решению.

## 7. Выбор вмешательства

### 7.1. Вмешательства по ребёнку (через наставника)

| Сигнал | Вмешательство бота |
|--------|-------------------|
| Level 0-1 после 8 эп | Предложить: упростить трек → A, знакомые контексты |
| Level 2, нет переноса | Предложить: 2-3 новых контекста для того же принципа |
| FM-02 (агентность) | Предложить: убрать вопросы-экзамены, вернуть выбор |
| FM-04 (перегруз) | Предложить: пауза 1 нед, потом ≤3 эп/нед |

### 7.2. Вмешательства по наставнику

| Сигнал | Вмешательство бота |
|--------|-------------------|
| Доля «с подсказкой» растёт | Напомнить: «паузу 10 сек, не подсказывай» |
| Нет рефлексии 2+ недели | Задать 3 вопроса рефлексии в чате |
| FM-05 (не учится) | Предложить: «объясни принцип своими словами» перед следующим скриптом |
| Логи без цитат | Напомнить: «запиши одну фразу ребёнка» |

### 7.3. Вмешательства по контуру

| Сигнал | Вмешательство бота |
|--------|-------------------|
| Перерыв >7 дней | Мягкое напоминание + «возвращаемся к знакомому контексту» |
| FM-03 (рассинхрон) | Предложить: сгенерировать памятку для со-наставника |
| Guard Policy пройден | Предложить: переход к следующему принципу + план |
| Готов ко второму витку | Предложить: углубление, мини-проект, новый формат |

### 7.4. Вмешательства по боту (авторы)

| Сигнал | Что делают авторы |
|--------|-----------------|
| Скрипты не используются | Пересмотреть промпт, добавить контекст |
| FM-detection ложноположительных >30% | Уточнить паттерны детекции |
| Dropout >50% за месяц | Пересмотреть онбординг, упростить вход |

## 8. Пилот / внедрение

### 8.1. Фазы

| Фаза | Что | Кто | Бюджет | Критерий перехода |
|------|-----|-----|--------|------------------|
| 0 | Этот документ (цепочка) | Автор | 3h | Решения приняты |
| 1 | MVP бота: онбординг + скрипт эпизода | Автор + AI | 6-8h | Один скрипт сгенерирован и проведён |
| 2 | + лог + анализ + Mastery Map | Автор + AI | 4-6h | 4 недели логов, карта заполнена |
| 3 | + FM-detection + Guard Policy + стратегия | Автор + AI | 4-6h | FM корректно детектируется |
| 4 | Тестеры (2-3 семьи из пилота) | Автор + тестеры | 2-3h | Обратная связь от 2+ семей |
| 5 | Ретроспектива → решение: выделять или интегрировать в AIST | Автор + Церен | 2h | Решение принято |

### 8.2. MVP (фаза 1): что входит

| Сценарий | Что делает бот | Что делает наставник |
|----------|---------------|---------------------|
| `/start` | Онбординг: собирает профиль семьи (диалог) | Отвечает на вопросы |
| `/episode` | Генерирует скрипт по: принцип + зона + трек + контекст | Проводит эпизод, записывает лог |
| `/log` | Принимает лог (структурированный ввод) | Вводит: контекст, цитата, сам/подсказка, трудность |
| `/status` | Показывает: текущий принцип, уровень, эпизодов за неделю | Ориентируется |

### 8.3. FSM-состояния (MVP)

```
IDLE
  → /start → ONBOARDING (сбор профиля)
    → завершён → IDLE

IDLE
  → /episode → EPISODE_PREP (выбор принципа/контекста)
    → сгенерирован → EPISODE_READY (показан скрипт)
    → «провёл» → LOG_INPUT (ввод лога)
    → записан → IDLE

IDLE
  → /log → LOG_INPUT (ввод лога без скрипта)
    → записан → IDLE

IDLE
  → /status → показать дашборд → IDLE
```

## 9. Ретроспектива → обновление норм

### 9.1. Ритм ретроспективы

| Когда | Что проверяем | Что обновляем |
|-------|-------------|--------------|
| После 4 недель MVP (фаза 1-2) | Скрипты адекватны? Лог-формат удобен? Наставник пользуется? | Промпты, формат ввода лога, FSM |
| После 8 недель (фаза 3) | FM-detection работает? Guard Policy корректен? | Паттерны детекции, пороги |
| После тестеров (фаза 4) | Онбординг понятен? Скрипты подходят другим семьям? | Онбординг, адаптация под разные семьи |
| Перед AIST (фаза 5) | Standalone или интеграция? Что переиспользовать? | Архитектурное решение |

### 9.2. Что обновляется по итогам

- Нормы (§3): пороги ритма, FM, качества логов — по реальным данным
- Индикаторы (§4): какие полезны, какие шум
- Промпты (DS-zpfp-family-guide/prompts/): уточняются по обратной связи
- Pack (PACK-kids-learning): если бот выявил новые FM или нужны новые карточки

## Открытые архитектурные вопросы

### OAQ-2. Digital twin: один или два?

**Предложение Церена:** детский ЦД = подпрофиль в родительском аккаунте.
**Решение для MVP:** один профиль семьи в SQLite с вложенными записями детей. Не digital twin в полном смысле — просто структурированное хранение. При интеграции в AIST — пересобрать под их модель ЦД.

### OAQ-3. Интеграция с WP-149 (Портной/Оценщик/Диагност)

**Решение для MVP:** standalone. Роли WP-149 реализуем как функции бота:
- Портной = `/episode` (генерация скрипта)
- Оценщик = `/log` → анализ (определение уровня)
- Диагност = FM-detection (по паттернам в логах)
- Навигатор = `/status` + Guard Policy

При интеграции в AIST — маппинг на их роли.

### OAQ-4. AI-генерация vs готовый контент

**Решение:** гибрид.
- Скрипт эпизода: **AI-генерация** (Claude Sonnet) с контекстом из Pack (ядро принципа, зона, трек). Промпт из `prompts/episode-script.md`.
- Анализ лога: **AI-анализ** (Claude Haiku) с правилами из Pack (уровни 0-3, FM-паттерны). Промпт из `prompts/log-analysis.md`.
- Стратегия: **AI-генерация** (Claude Sonnet) с контекстом семьи + Pack. Промпт из `prompts/strategy-generation.md`.
- Guard Policy: **правила** (хардкод): Level 3 ≥3 раз, ≥2 контекста, спонтанно.
- FM-detection: **гибрид** — правила (счётчики в логах) + AI (интерпретация).
